# Hermes Agent セキュリティポリシー

このドキュメントでは、**Hermes Agent** プロジェクトのセキュリティプロトコル、信頼モデル、デプロイ強化ガイドラインを概説します。

> **注意:** 完全な英語版は [SECURITY.md](SECURITY.md) をご覧ください。

## 1. 脆弱性の報告

Hermes Agentはバグ報奨金プログラムを運営していません。セキュリティの問題は [GitHub Security Advisories (GHSA)](https://github.com/NousResearch/hermes-agent/security/advisories/new) または **security@nousresearch.com** へのメールで報告してください。セキュリティ脆弱性について公開Issueを作成しないでください。

### 必要な提出情報
- **タイトルと重要度:** 簡潔な説明とCVSSスコア/レーティング
- **影響を受けるコンポーネント:** 正確なファイルパスと行範囲（例：`tools/approval.py:120-145`）
- **環境:** `hermes version`の出力、コミットSHA、OS、Pythonバージョン
- **再現方法:** `main`または最新リリースに対するステップバイステップのPoC
- **影響:** どの信頼境界が越えられたかの説明

---

## 2. 信頼モデル

コアの前提は、Hermesが**一人の信頼されたオペレーター**を持つ**パーソナルエージェント**であることです。

### オペレーターとセッションの信頼
- **シングルテナント:** システムはLLMアクションからオペレーターを保護しますが、悪意のある共同テナントからは保護しません。マルチユーザー分離はOS/ホストレベルで行う必要があります。
- **ゲートウェイセキュリティ:** 認可された呼び出し元（Telegram、Discord、Slackなど）は同等の信頼を受けます。
- **実行:** デフォルトは`terminal.backend: local`（直接ホスト実行）。コンテナ分離（Docker、Modal、Daytona）はサンドボックス化のためにオプトインです。

### 危険なコマンドの承認
承認システム（`tools/approval.py`）はコアのセキュリティ境界です。ターミナルコマンド、ファイル操作、その他の潜在的に破壊的なアクションは、実行前に明示的なユーザー確認によってゲートされます。

### スキルとMCPサーバー
- **インストール済みスキル:** 高い信頼。ローカルホストコードと同等です。
- **MCPサーバー:** 低い信頼。MCPサブプロセスはフィルタリングされた環境を受け取ります。

---

## 3. スコープ外（非脆弱性）

以下のシナリオはセキュリティ違反とは見なされません：
- 承認システムの具体的なバイパスにつながらない**プロンプトインジェクション**
- 外部認証なしで公開インターネットにゲートウェイを公開すること
- `~/.hermes/`、`.env`、`config.yaml`への事前書き込みアクセスが必要なレポート
- `terminal.backend`が`local`の場合のホストレベルコマンド実行（ドキュメント化されたデフォルト）

## 4. デプロイ強化とベストプラクティス

### ファイルシステムとネットワーク
- **本番サンドボックス化:** 信頼できないワークロードにはコンテナバックエンド（`docker`、`modal`、`daytona`）を使用
- **ファイル権限:** non-rootで実行；ローカルインストールでは`chmod 600 ~/.hermes/.env`で資格情報を保護
- **ネットワーク公開:** VPN、Tailscale、またはファイアウォール保護なしでゲートウェイやAPIサーバーを公開インターネットに晒さないでください

### スキルとサプライチェーン
- **スキルインストール:** サードパーティスキルをインストールする前にSkills Guardレポートを確認
- **MCP安全性:** `npx`/`uvx`パッケージに対してOSVマルウェアチェックが自動実行
- **CI/CD:** GitHub Actionsは完全なコミットSHAに固定

### 資格情報の保管
- APIキーとトークンは`~/.hermes/.env`にのみ保管 — `config.yaml`やバージョン管理にはコミットしない

## 5. 開示プロセス

- **協調的開示:** 90日間の窓口、または修正がリリースされるまで（いずれか早い方）
- **コミュニケーション:** すべての更新はGHSAスレッドまたはsecurity@nousresearch.comとのメールで行われます
- **クレジット:** 匿名希望がない限り、レポーターはリリースノートにクレジットされます
